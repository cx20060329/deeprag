# Progress Update — 2026-06-18

## Completed Tasks

### 1. BGE-M3 Embedding Upgrade + Reranker Enable
- Modified `retrieval/embedder.py`: BGE-M3 now priority model (1024-dim), local path resolution with weight file validation
- Modified `retrieval/reranker.py`: Cross-Encoder now loads properly with local path check, `_resolve_model_path()` validates weight files exist
- Modified `retrieval/pipeline.py` Stage 6: Cross-Encoder enabled by default (no longer gated by `_quality_mode`)
- **Key change**: Embedder and reranker now check local `models/BAAI/` directory first with `local_files_only=True`, fall back to HF Hub if unavailable
- **Note**: Local model weight files (`pytorch_model.bin`, `model.safetensors`) were corrupted from original download. Removed. Models will download from HF Hub on first use. If HF inaccessible, code falls back to next available model.

**Why:** BGE-M3 (1024d) provides better semantic quality than bge-small-zh-v1.5 (512d). BGE Reranker v2 M3 cross-encoder provides much better reranking than Jaccard.

### 2. TRANSITION_TO Relationship Fix
- Modified `content_analysis/table_analyzer.py`:
  - `TRANSITION` schema: Added `"转移"` to `source_state` keywords, added `transition_text` field
  - Header groups tightened: removed `"源"` and `"目标"` (too broad, caused false positive with `源通道/目标通道` tables)
  - New `_split_transition_records()` method: splits `"OFF -> ShortTermLighting"` into source/target
  - Enhanced `_rel_transition()`: uses `transition_text` field for better Format 2 inference
- Modified `content_analysis/state_machine.py`: `TRANSITIONS_TO` → `TRANSITION_TO` (consistent with models.py)
- **Result**: TRANSITION_TO went from **0 → 34** relationships across InteriorLight and ATWS modules

**Why:** BCM transition tables use `序号 | 转移 | 事件 [说明]` format where source→target states are combined in one column with `->`. Previous code failed to parse this format.

### 3. Neo4j + Qdrant Runtime Deployment
- Added `neo4j>=5.0` and `qdrant-client>=1.0` to `pyproject.toml`
- Created `retrieval/graph_store.py`: `BaseGraphStore` ABC with `NetworkXGraphStore` (memory) and `Neo4jGraphStore` (production)
- Created `retrieval/vector_store.py`: `BaseVectorStore` ABC with `NumpyVectorStore` (memory) and `QdrantVectorStore` (production)
- Created `retrieval/neo4j_config.py` and `retrieval/qdrant_config.py` for env-based configuration
- Updated `retrieval/graph_retriever.py`: delegates to `BaseGraphStore`, backward compatible
- Updated `retrieval/dense_retriever.py`: delegates to `BaseVectorStore`, backward compatible
- Fixed `retrieval/pipeline.py`: removed duplicate `import json` that shadowed global
- Created `scripts/import_to_neo4j.py` and `scripts/import_to_qdrant.py`
- **Default behavior unchanged**: memory backends used unless `BCM_USE_NEO4J=1` / `BCM_USE_QDRANT=1`

**Why:** Production deployment needs persistent, scalable storage. The abstraction layer enables seamless switching between dev (in-memory) and production (Neo4j/Qdrant) modes via environment variables.
